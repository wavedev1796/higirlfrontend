# HG-F3-007 - Cliente HTTP centralizado

## Resumen

Se implemento un cliente HTTP unico para que todas las solicitudes del
frontend compartan la URL base, el token de autenticacion, la serializacion de
datos y el manejo de errores.

## Objetivo

Reemplazar las llamadas HTTP dispersas por un wrapper tipado de `fetch` con:

- token Bearer automatico;
- errores uniformes;
- URL base configurable;
- soporte para JSON;
- soporte para `FormData`.

## Implementacion

El cliente principal se encuentra en:

```text
src/lib/api.ts
```

Expone las siguientes funciones:

```text
api<T>()
apiClient.get<T>()
apiClient.post<T>()
apiClient.patch<T>()
apiClient.put<T>()
apiClient.delete<T>()
```

Los servicios de cada feature consumen `apiClient` y no llaman directamente a
`fetch`.

## URL base configurable

Las variables estan documentadas en `.env.example`:

```env
NEXT_PUBLIC_API_URL=http://localhost:3001/api/v1
NEXT_PUBLIC_API_ORIGIN=http://localhost:3001
```

Uso:

- `NEXT_PUBLIC_API_URL`: base de los endpoints REST.
- `NEXT_PUBLIC_API_ORIGIN`: origen publico para archivos y fotos.

La configuracion se centraliza en:

```text
src/infrastructure/config/env.ts
```

## Token automatico

Antes de cada solicitud, el wrapper consulta:

```text
src/infrastructure/storage/tokenStorage.ts
```

Si existe un token, agrega:

```http
Authorization: Bearer TOKEN
```

Los componentes y servicios no necesitan construir el header de
autenticacion.

## Manejo de respuestas 401

Cuando el backend responde `401`, el cliente elimina el token almacenado. Esto
evita conservar una sesion expirada o invalida.

La proteccion de rutas detecta posteriormente la ausencia de autenticacion y
redirige al login.

## Manejo uniforme de errores

Los errores se transforman en una instancia de `ApiError` con:

```text
message
status
data
```

El parser reconoce las formas utilizadas por el backend:

```text
message: string
message: string[]
mensaje: string
```

Si el backend no entrega un mensaje valido, se utiliza:

```text
No se pudo completar la solicitud.
```

## Solicitudes JSON

Cuando el cuerpo no es `FormData`, el wrapper agrega automaticamente:

```http
Content-Type: application/json
```

Los metodos `post`, `patch` y `put` serializan los objetos mediante
`JSON.stringify`.

## Solicitudes multipart

Cuando el cuerpo es una instancia de `FormData`, el wrapper no asigna
manualmente el `Content-Type`.

Esto permite que el navegador genere correctamente:

```http
Content-Type: multipart/form-data; boundary=...
```

Este comportamiento se utiliza para la foto de perfil.

Las rutas del backend se definen en:

- [x] Los errores se manejan de forma uniforme.
