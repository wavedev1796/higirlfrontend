# HG-F3-006 - Frontend Perfil

## Resumen

Se implementaron las pantallas de visualizacion y edicion del perfil de la
usuaria, incluyendo foto, diseno responsive, persistencia de sesion y
onboarding posterior al registro.

## Objetivo

Permitir que una usuaria autenticada consulte y actualice su perfil, incluyendo
una foto, y dirigir a las cuentas nuevas a completar sus datos.

## Rutas

| Ruta | Funcion |
|---|---|
| `/profile` | Visualizacion de los datos del perfil |
| `/profile/edit` | Edicion del perfil |
| `/profile/edit?onboarding=1` | Edicion con mensaje de onboarding |

## Pantalla Mi perfil

La pantalla muestra:

- nombre y apellido;
- nombre de usuario;
- correo;
- ciudad;
- fecha de nacimiento;
- profesion;
- empresa;
- estado civil;
- hijos y numero de hijos;
- biografia;
- foto;
- intereses y categorias devueltos por el backend.

Cuando no existe una foto se muestran las iniciales de la usuaria.
Cuando un campo opcional no tiene valor se muestra `Sin completar`.

## Pantalla Editar perfil

La pantalla permite modificar los campos aceptados por el DTO del backend:

```text
ciudad
fechaNacimiento
profesion
empresa
estadoCivil
tieneHijos
numeroHijos
bio
```

El nombre, correo y nombre de usuario son datos protegidos. Se muestran como
informacion de contexto, pero no se incluyen en el `PATCH`.

Si `tieneHijos` es falso, el frontend envia `numeroHijos` con valor `0`.

## Foto de perfil

La foto se selecciona mediante un input de archivo y se muestra inmediatamente
en una previsualizacion local.

Validaciones aplicadas:

- formatos JPEG, PNG y WebP;
- tamano maximo de 5 MB.

La subida usa:

```http
POST /api/v1/perfil/me/foto
Content-Type: multipart/form-data
Campo: foto
```

La eliminacion usa:

```http
DELETE /api/v1/perfil/me/foto
```

## Onboarding posterior al registro

Despues de un registro exitoso:

1. El frontend inicia sesion con las credenciales registradas.
2. Guarda el token mediante `AuthProvider`.
3. Redirige a `/profile/edit?onboarding=1`.
4. Muestra el bloque `Paso final`.
5. Permite completar los datos y la foto.
6. Al guardar, redirige a `/profile`.

Si la cuenta se crea pero el login posterior no entrega un token, se redirige
al login conservando la ruta de onboarding como `callbackUrl`.

## Sesion persistente

Al cargar la aplicacion, `AuthProvider` revisa si existe un token y consulta:

```http
GET /api/v1/perfil/me
```

Si el token es valido, reconstruye el estado de la usuaria.
Si es invalido, elimina el token y deja la sesion cerrada.

## Proteccion de rutas

El layout privado utiliza `ProtectedRoute`.

Cuando no existe una sesion valida se redirige a:

```text
/login?callbackUrl=RUTA_SOLICITADA
```

Despues del login se respeta `callbackUrl`, siempre que sea una ruta interna.

## Integracion con el backend

Endpoints utilizados:

| Metodo | Endpoint | Funcion |
|---|---|---|
| `POST` | `/auth/register` | Crear cuenta |
| `POST` | `/auth/login` | Iniciar sesion despues del registro |
| `GET` | `/perfil/me` | Consultar el perfil |
| `PATCH` | `/perfil/me` | Actualizar los datos |
| `POST` | `/perfil/me/foto` | Subir o reemplazar la foto |
| `DELETE` | `/perfil/me/foto` | Eliminar la foto |

El frontend dejo de enviar el campo `rol` durante el registro. El DTO actual
del backend rechaza campos no permitidos y el backend ya asigna `user`.
