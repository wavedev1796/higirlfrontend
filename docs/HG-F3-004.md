# HG-F3-004 - Refactorización Arquitectónica SOLID y Estructura por Features

## Resumen

Se realizó una refactorización integral del frontend para migrar de una estructura plana basada en archivos a una arquitectura modular orientada a **dominios (features)** y siguiendo los principios **SOLID**. Esta nueva estructura mejora la mantenibilidad, escalabilidad y testabilidad del proyecto.

## Objetivo

Organizar el código de manera que cada módulo tenga una responsabilidad única, separando la lógica de negocio (servicios/hooks) de la interfaz de usuario (componentes) y el acceso a datos (servicios API).

## Alcance Implementado

- **Estructura por Features**: Creación de `src/features/auth` para encapsular toda la lógica de autenticación.
- **Principio de Responsabilidad Única (SRP)**:
  - **Servicios**: `authService.ts` maneja exclusivamente la comunicación con el backend.
  - **Hooks**: `useLogin` y `useRegister` manejan el estado y efectos secundarios de las peticiones.
  - **Componentes**: `LoginForm` y `RegisterForm` se enfocan únicamente en la presentación y captura de datos.
- **Abstracción de API**: Implementación de `src/lib/api.ts` como un wrapper genérico y tipado para `fetch`, centralizando el manejo de tokens y errores.
- **Componentes de UI Reutilizables**: Creación de `src/components/ui` (`Button`, `Input`) para promover la consistencia visual y seguir el principio **Open/Closed**.
- **Gestión de Rutas y Grupos**:
  - Uso de grupos de rutas Next.js: `(auth)` para autenticación y `(dashboard)` para áreas protegidas.
  - Centralización de rutas en `src/constants/routes.ts`.
- **Estado Global**: Implementación de `AuthProvider` en `src/store/authStore.tsx` para gestionar la sesión del usuario en toda la aplicación.

## Bugs Encontrados y Corregidos

### BUG 1: URL de API sin prefijo `/api/v1`
- **Archivo**: `.env.local`
- **Problema**: `NEXT_PUBLIC_API_URL` estaba configurado como `http://localhost:3001` sin el prefijo global `/api/v1` que el backend NestJS define con `app.setGlobalPrefix('api/v1')`. Todas las peticiones del frontend iban a rutas inexistentes (ej: `http://localhost:3001/auth/login` en lugar de `http://localhost:3001/api/v1/auth/login`).
- **Corrección**: Cambiado a `http://localhost:3001/api/v1`.

### BUG 2: CORS apuntando al puerto incorrecto
- **Archivos**: `higirlbackend/src/main.ts`, `higirlbackend/src/config/app.config.ts`
- **Problema**: El origin de CORS estaba configurado como `http://localhost:3001` (el puerto del propio backend). El frontend Next.js corre en el puerto `3000`, por lo que las peticiones desde el navegador serían bloqueadas por CORS.
- **Corrección**: Cambiado el default de `FRONTEND_URL` a `http://localhost:3000` en ambos archivos.

### BUG 3: Login no integrado con AuthProvider ni redirección
- **Archivo**: `src/features/auth/components/LoginForm.tsx`
- **Problema**: El `LoginForm` usaba el hook `useLogin` para hacer la petición, pero no integraba con el `AuthProvider` (`useAuthStore`) para actualizar el estado global del usuario. Además, no había redirección al dashboard después de un login exitoso. El usuario se quedaba en la pantalla de login indefinidamente.
- **Corrección**: 
  - Se importó `useRouter` y `useAuthStore`.
  - Después de un login exitoso, se llama a `authStore.login()` con los datos del usuario y el token.
  - Se redirige automáticamente a `ROUTES.DASHBOARD` (`/feed`).
  - Se agregó condición `!error` para no mostrar el preview de sesión cuando hay error.
